BCompiler Script File Reference

(all-u-want-2-know about Data.src)

 

 

Data.src format:

 

Data.src is a plain text file that when compiled creates MuData.lib. The format of the text is very simple. It's a line-based script. Each line means an insertion in the database. Because the file keeps various types of information, the first character of the line determines this type.

 

For lines started with:

 

#

 

Inserts a record into the ItemType list.

 

There are two syntaxes for this command:

 

'#X,TypeName' and '#X,Y,Z,TypeName'

 

Where X is the class number of the item. In the first case, every item that has the class number equals to X will be classified as 'TypeName'.

     

Y and Z defines a range within a class. This range means that every item of class X between IDs Y and Z will be classified as ‘TypeName’.

 

e.g.:

 

#12,3,6,Wings lvl2

 

In this example, items with ClassNumber equal to 12 and ClassID between 3 and 6 will be classified as 'Wings lvl2'.

 

~

Defines the direction of a determined index in the direction descriptor.

 

This command has little use, because the direction indexes are supposed to be the same for different versions of Mu.

 

e.g.:

 

~1, nw

 

The example above means that index 1 will be assigned to direction north-west.

 

@

Sets a spot.

 

This operator is the most useful, I think. You can name spots to be used in CharBlaster. Can be used to set the spawn-points or even the NPCs inside the city.

 

e.g.:

 

@0,133,118,Lorencia

 

Here the 'Lorencia' spot will keep the information of map #0 (Lorencia), in X-Axis 133 and Y-Axis 118. Note that the name of the spot shown (Lorencia) can be different of the map name.

 

e.g.:

 

@0,133,118,Devias

 

It'll only change the spot name. The map will be Lorencia (map #0) anyway.

 

%

Adds a class to classlist.

 

This command, like the '~', has little use. It adds a classname (e.g. 'DarkLord') assigning it to a classnumber (e.g. 64).

 

e.g.:

 

%64,Dark Lord

Any other lines (except the blank ones) will be considered item assignments. The syntax for item assignment is:

 

Class,ID,ItemName,X,Y,HasLvl,Lvl

 

Explaining parameters... but first, let me say something about the MuOnline Item Classification Method.

 

Every item in MuOnline has a Class and an ID. For example, Short Sword has the class 0 (used to be the single blades class), and ID 1 (after Kris, that has the ID 0). Simple, huh? But there is a problem.

 

Every item has a Class and an ID, but not every pair Class-ID describes an item. It's better shown with the Boxes. Box Of Luck, Heart Of Love, Box of Kundum, etc. are 'same item'. All have the class 14 and ID 11. I prefer to call this item as Generic Box. What makes the difference between them? The level. The generic box level 0 is the Box Of Luck, the same way as the generic box level 12 is the Box Of Kundum '+5'.

 

As you can see, some items are classified by their levels. It's all I wanted to say. Now, lets return to the parameters.

 

Class

The class of the item

ID

The ID of the item

ItemName

The name that item will assume in program.

X, Y

The 'width' and 'height' of the item.

HasLvl

Must to be ‘1’ when the item is level-classified

Lvl

If the item is level-classified, specifies which level it must be

    

e.g.:

 

14,11,Gold Medal,1,1,1,6

 

Assign the 'Gold Medal' name to item with class 14, id 11 when it's on level 6.

 

Note:

When an item doesn't have level-classifying, all levels (0 to 15) are assigned to same name specified on the command.

 

e.g.:

 

13,14,Blue Feather,1,2,0,0

 

'The item class13 id14 in every level will be named Blue Feather' is what this line says.

 

Important:

The source script file (data.src, in this case) is REALLY a script file. It means that the file is parsed line by line and you must consider it while editing. For example, if you insert an item definition BEFORE a class definition ('#') you have a chance to get your item unclassified.

 

e.g.:

 

13,14,Blue Feather,1,2,0,0

.

.

.

#13,Feather

 

If it happens on your script file, Blue Feather won't be classified as a Feather, but as any other class specified before (or unclassified, if no class has been specified before).
